# Devin検証メモ（移送オーダ一覧 OFFSITE-V-0401）

## ■ 概要
- Excel画面設計書を元にDevinで業務画面を実装
- テスト実施および報告書作成まで一連の流れを検証
- プロンプトは段階的に投入（実装 → テスト → 修正）

---

## ■ 1回目（実装）
設計書をDevinに投げる+以下

### 制約
- 単一画面のみ実装（他画面禁止）
- 画面遷移はダミー
- 業務機能の追加は禁止
  - 新しいステータス
  - 新しい検索条件
  - 新しいボタン
  - 承認 / 通知 / バッチ / 帳票 / 外部連携

### データ
- SQLiteの仮DBをローカルに作成
- 初期データ投入（約20件）
- 以下を含める
  - ステータス違い
  - 日付違い
  - 空値
  - 長文
  - マスタ未設定
  - 0件検索条件

### 成果物
- フロントエンド
- バックエンドAPI
- DB / seedデータ
- README

### 結果
- 指示通り生成（暴走なし）
- UI / API / DBが一貫して生成
- UIはやや古めの業務画面

## ■ 総評


### ■ プロンプト設計に関する知見
- Devin用プロンプトはそのまま使用するのではなく、事前にCopilot等で整理・構造化することで精度が向上する
- プロンプトは分割して段階投入するよりも、初回に制約・成果物・禁止事項・完了条件まで網羅的に記述した方がACU削減につながる
- 小出しの修正指示は往復回数が増え、結果的にコスト増となる傾向がある
- Devin活用においては「実装前のプロンプト設計」が生産性を大きく左右する

---

### ■ 設計書との関係
- 設計書はそのままだとAIにとって読みづらいケースが多く、構造化・不要情報削除・表記統一などの前処理が必要になるはず。
- Devinの出力品質はプロンプトだけでなく、入力となる設計書の粒度・整合性・記述品質に大きく依存する
- Excel設計書でも対応可能だが、セル結合・略語・注記分散などがあると誤読の要因となる
- 理想は「顧客にも読みやすく、かつDevinにも読みやすい設計書」であり、AI活用を前提とした設計書標準の見直し余地がある
---

### ■ 適用可能領域（強み）
- 画面単位の実装（一覧・検索・参照など定型UI）
- 仮DB・seedデータの生成
- モックAPIの作成
- READMEや報告書など周辺成果物の生成
- UI / API の一貫した初期実装
- 試作・たたき台の高速作成

---

### ■ 開発における位置づけ
- Devinは要件定義や設計を代替するものではなく、設計が固まった後の実装を加速するツールである。（ウォーターフォール型においては）
- 基本的にアジャイル向きな製品であると感じた。
- 要件・制約・入出力が明確な場合に性能を発揮する
- 初期画面を短時間で可視化できるため、顧客レビューの前倒しや認識齟齬の早期解消に有効
---

### ■ 制約・注意点（弱み）
- UIデザインはやや旧来の業務システム寄りとなる傾向がある
- 添付ではなくリポジトリ保存を指定するなど、出力方法の明示が必要
- 一見正しく動作するが、業務要件を完全に満たしているとは限らない

---

### ■ 非機能要件に関する考察
非機能要件（性能、セキュリティ、トランザクション制御、障害対応、運用性等）を含む領域については、以下の傾向がある。

- 動作するコードは生成可能であるが、本番利用を前提とした品質担保は別途設計および検証が必要
- 特に以下の領域は注意が必要
  - 性能（同時接続、レスポンス、データ量）
  - セキュリティ（認証・認可、脆弱性対策）
  - トランザクション制御（整合性、排他制御）
  - 障害対応（リトライ、ログ、監視）
  - 運用性（設定管理、保守性、拡張性）

Devin単体でこれらを担保するのは難しく、人間による設計・レビューが前提となる（？）

---

### ■ バックエンドに関する所感
- 本検証ではバックエンドの高度な業務ロジックは未検証
- 一般的なCRUDや単純な処理は対応可能と考えられる
- 一方で、複雑な業務判定、トランザクション制御、非機能要件を含む処理は別途検証が必要

---

### ■ 運用上のベストプラクティス
- プロンプトは初回に最大限詳細に記述する
- 成果物の保存先・形式・命名まで明示する
- 設計書はAI前提で整形する
- 画面単位で生成し、逐次レビューする
- 生成物は必ず人間がレビューする　

---

### ■ アンチパターン（避けるべき使い方）
- 要件が曖昧なまま丸投げする
- 設計書を未整理のまま投入する
- 小さな修正を何度も往復する
- 生成物をレビューせずそのまま採用する
- 非機能要件を考慮せず本番利用する

---

### ■ 結論
Devinは、適切なプロンプト設計と設計書前処理を前提とすれば、業務画面の初期実装および試作作成において非常に高い生産性を発揮する。
一方で、非機能要件や複雑な業務ロジックを含む領域については人間による設計・レビューが不可欠であり、単独で開発を完結させるツールではないはず。
